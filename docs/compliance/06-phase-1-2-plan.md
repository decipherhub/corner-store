# Phase 1~2 실행 플랜 — 데모 완성까지 우리가 할 일 (v2)

> **이 문서 하나로 충분하도록 썼습니다.** 다른 문서를 안 읽어도 ① 우리가 뭘 만들고 있고 ② "완성"이 뭔지 ③ 지금 무엇을 결정·리서치·구현해야 하는지 ④ 그것이 어떻게 돌아가는 모습으로 끝나는지를 알 수 있습니다. (괄호 속 작은 코드들은 상세 문서 추적용 — 무시하고 읽어도 됩니다.)

## 1. 우리가 만들고 있는 것 — 두 문단 요약

우리는 **"규제를 코드로 검사하는 거래 인프라"**를 만들고 있습니다. 증권 토큰은 아무나 사고팔 수 없습니다 — 매수인 자격, 매도인의 보유 기간, 제재 명단, 시장행위 규제 같은 법적 요건이 거래마다 붙습니다. 우리 시스템은 이 요건들을 **부품(Element) → 법률 단위 묶음(Recipe) → 자산별 신상카드(Manifest) → 운영 주체(Operator)**의 4층으로 코드화해서, *위법한 거래는 체결 자체가 안 되게* 만듭니다. 제품의 주력은 이 **컴플라이언스 표준·SDK**이고, 우리가 만드는 DEX(Corner Store)는 그것이 작동함을 보여주는 시연장입니다.

**Phase 2의 목표는 데모 하나입니다**: BlackRock의 토큰화 펀드 **BUIDL**(기관전용 사모 MMF의 토큰판 — 자격 있는 대형 투자자만 살 수 있음)이 ERC-3643 표준 토큰으로 존재한다고 가정하고, 그것이 **우리 DEX에서 모든 법적 검사를 통과하며 거래되는 모습을 처음부터 끝까지 시연**하는 것. 미국에서 실제 영업 인가를 받는 일(중개업 등록 등)은 Phase 3 — 이번 범위가 아닙니다.

## 2. "완성"의 정의 — 데모에서 보여줄 6 장면

이 6 장면이 돌아가면 Phase 2는 끝난 것입니다. 각 장면은 발표 데모이자, 규제 논증 자료이자, 나중에 미국 인가 신청 서류의 원형으로 — 한 번 만들어 세 번 씁니다.

| # | 장면 | 무슨 일이 일어나나 | 무엇을 증명하나 |
|---|------|------------------|----------------|
| 1 | **정상 거래** | 자격 있는 기관 A가 기관 B에게 BUIDL을 판다 → 모든 검사 통과 → 한 번의 트랜잭션으로 체결 | 시스템 전체가 맞물려 돈다 |
| 2 | **거절 ①** | 자격 없는(기관전용 기준 미달) 투자자가 사려고 한다 → 체결 거부 + 사유 기록 | 투자자 자격 규제가 코드로 작동 |
| 3 | **거절 ②** | 발행사(BlackRock) 계열 계정이 자기 토큰을 사려고 한다 → 거부 | "판매 중인 증권을 파는 쪽이 사들이면 안 된다"는 시세 관여 금지 규제가 작동 |
| 4 | **거절 ③** | 자격 증명이 만료된 보유자가 거래를 시도한다 → 거부 + 재인증 안내 | 자격에 유효기간이 있음을 시스템이 안다 |
| 5 | **규칙 갱신** | 제재 명단이 바뀐다 → 다중 서명 승인 → **거래 중단 없이** 새 규칙 반영 | "새 규제 대응 = 코드 수정이 아니라 데이터 등록"이라는 핵심 주장 |
| 6 | **시장 감시** | 같은 사람의 두 지갑이 자전거래를 한다 → 체결은 되지만 자동으로 표시(flag)되어 운영팀 검토 큐에 들어간다 | 막을 수 없는 영역은 감시하고 기록한다 |

## 3. 일정 — 2주 × 4 스프린트

| 스프린트 | 이름 | 핵심 |
|---------|------|------|
| 1 (지금) | **결정하고 쏘기** | 쌓인 결정들을 처리하고(§4), 개발팀에 수정 요청을 보내고, 공유 문서 패키지와 변호사 질문지를 발송 |
| 2 | **데이터와 연결** | BUIDL 신상카드 작성(코드 아닌 데이터 작업) + 발행사 쪽 자격 증명을 신뢰해 받아오는 연결 |
| 3 | **검사와 통로** | 검사 부품들 연결 + 신규 부품 1개 개발 + "청약·환매 창구"와 "DEX 매매"의 기술적 분리 |
| 4 | **시연과 논증** | 6 장면 데모 실행 + BUIDL 규제 논증서(법→논리→코드 추적 문서) 작성 + SDK 문서 골격 |

**일정의 진짜 병목은 리서치도 개발도 아니라 §4의 결정들입니다.** 변호사 회신은 일정에 영향을 주지 않습니다 — 회신 전에는 보수적 기본값으로 차단해 두고, 회신·규제 완화가 오면 *설정값 하나만 바꿔서* 반영하도록 설계되어 있기 때문입니다.

## 4. 지금 결정해야 할 것들 — 평문으로

### 4-가. 설계 확정 4건 (규제 담당 결정 — 처리 즉시 "부품 목록 확정" 선언)

| # | 질문 | 배경 | 권고 |
|---|------|------|------|
| 1 | **"판매 중인 토큰을 발행자 쪽이 사들이는 것을 막는 검사 부품"을 정식 추가할까?** | BUIDL 같은 펀드 토큰은 매일 신규 발행 중 = 법적으로 "판매가 끝나지 않은 상태". 이때 발행사·계열사가 시장에서 자기 토큰을 사면 시세 관여 금지 규제(미국 Reg M — 한국의 안정조작 규제 사촌) 위반 소지. 이걸 막는 부품의 설계는 끝났고 추가 여부만 남음 (F-04) | **추가** |
| 2 | **공매도 규제 부품은 설계 메모만 남기고 코드는 안 만들까?** | 우리 DEX엔 공매도 기능이 없어서 지금은 불필요. 나중에 도입하면 그때 만들면 됨 (F-05) | **메모만** |
| 3 | **제재 명단·국가 제한 검사를 "증권법 묶음"에서 독립시킬까?** | 제재(OFAC)·국가 제한은 증권법이 아니라 *별개 법률*이고 증권이 아닌 자산에도 적용됨. 지금은 증권법 묶음 안에 끼어 있는데, 독립 묶음으로 빼면 다른 자산·다른 나라로 확장할 때 그대로 재사용 가능 (OD-CI-5 — 횡단 Recipe) | **독립** |
| 4 | **부속 문서 2건의 한 줄 보완을 지금 할까?** | 분류 체계 문서에 "항상 켜져 있는 부품" 구분 한 줄 + 논증서 양식에 "운영 의무" 열 추가 — 30분 작업 (OD-CI-4) | **실행** |

| #   | 질문                                                            | 왜 필요한가                                                                            |
| --- | ------------------------------------------------------------- | --------------------------------------------------------------------------------- |

## 5. 리서치·법률 트랙 — 무엇을 더 알아보나

1. **사실 확인 1건 (30분)**: BUIDL의 월 이자가 *신규 토큰 지급*인지 *잔액 자동 증가*인지 — 공식 문서 확인. §4-나-6의 설계가 여기에 달림
2. **변호사 질문지 발송**: 그동안 쌓인 질문들(중개업 등록 경계·시세 관여 규제 적용·역외 적용·자격자 간 전매 근거 등)을 한 통으로 통합해 발송 준비 끝. *발송해도 일정은 안 막힘* (§3 설명)
3. **규제기관 사전 질의 전략 문서**: "기준가($1 고정) 기반 펀드 토큰은 시세 관여가 원천 불가능하니 매수 금지를 풀어달라"는 논리로 미국판 비조치의견서를 받는 전략 — 초안 논거 4종은 이미 정리됨
4. (병행 학습) 미국 사기·내부자거래 법리 — 시장 감시 부품의 법적 기초

## 6. 구현 트랙 — 개발팀과 무엇을 만드나

> 우리 쪽은 *수정 요청서* 형태로 전달하고, 채택·구현 방식의 결정 권한은 개발팀에 있습니다.

| 단계  | 작업                                                                                                        | 성격  |
| --- | --------------------------------------------------------------------------------------------------------- | --- |
| A   | **BUIDL 신상카드 등록** — 이 자산에 어떤 법이 적용되고(기관전용 펀드·사모 발행·상시 판매 중) 어떤 명단·기준값이 붙는지를 데이터로 등록                       | 데이터 |
| B   | **발행사 쪽 자격 증명 연결** — 발행사 측(명의개서대행)이 이미 확인한 "이 투자자는 자격 있음" 증명을 우리가 재검증 없이 받아 쓰되, *무엇을 신뢰했는지 기록*            | 연동  |
| C   | **검사 부품 연결 + 신규 부품 1개 개발** — 기존 부품 27종 연결 + §4-가-1의 신규 부품                                                 | 코드  |
| D   | **창구 분리** — 청약·환매(발행사 창구)와 DEX 매매를 기술적으로 다른 통로로. 환매까지 막으면 안 되고, 환매가 시장 매수로 오해받아도 안 되기 때문                  | 코드  |
| E   | **6 장면 시연** (§2) + 거래 기록이 규제 보존 양식으로 내보내지는지 확인                                                            | 검증  |
| F   | **BUIDL 규제 논증서 작성** — "어느 법이 → 어떤 논리로 → 어느 코드에" 연결되는지의 추적 문서. 변호사의 법률검토보고서를 시스템이 실시간 실행하고 있음을 보여주는 최종 산출물 | 문서  |
