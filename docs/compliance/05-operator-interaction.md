# [팀 공유] Operator(운영) 레이어는 시스템과 어떻게 만나는가 (구체판)

> 우리 시스템의 4개 층 — Element(요건 검사 부품) / Recipe(법률 단위 검사 묶음) / Manifest(자산별 신상카드) / **Operator(운영 주체)** — 중에서, Operator가 나머지와 어떻게 상호작용하는지의 정리입니다. 새로운 결정은 없고 그동안 합의된 내용의 통합입니다. §1~3은 개념, §4~5는 구체 규칙, §6은 함께 결정할 것들입니다.

## 1. 핵심 명제 한 줄

> **"Operator는 체결 경로(hot path)에 없다 — 상태와 거버넌스로만 말한다. 그리고 모든 발화는 서명되고 기록된다."**

오해 방지 두 가지:
- **매칭 엔진이 없다는 뜻이 아닙니다.** 엔진은 컨트랙트로 존재하고 자율 실행됩니다. *운영자가 개별 체결에 재량으로 개입할 수 있는 호출 경로가 설계상 존재하지 않는다*는 뜻입니다.
- **운영자가 무력하다는 뜻도 아닙니다.** 운영자는 세 개의 강력한 통로(사전 심사·상태 입력·사후 처리)를 갖습니다 — 다만 그 통로 전부가 *권한이 제한되고(modifier), 여럿의 서명이 필요하고(multi-sig), 기록이 남는* 방식일 뿐입니다.

## 2. 전체 그림 — 두 개의 평면

시스템은 **거버넌스 평면**(규칙을 만들고 바꾸는 곳 — 느리고 신중)과 **runtime 평면**(거래가 체결되는 곳 — 빠르고 자동)으로 나뉘고, Operator는 거버넌스 평면에만 삽니다:

```
        [ 거버넌스 평면 — 사전 (느림·신중) ]          [ runtime 평면 — 체결 경로 (빠름·자동) ]
                                                   ┌──────────────────────────────────┐
  Operator ──심사·승인 (다중서명+시차)──▶ Recipe       │  엔진: 신상카드 조회                  │
  Operator ──온보딩 검증·등록──▶ Manifest(신상카드) ───▶│      → 적용 Recipe 조합              │
  Operator ──부품 버전업 심사──▶ Element              │      → 부품 일괄 판정                │
                                                   │      → 전부 통과 시 체결 / 아니면      │
        [ 상태 채널 — "판단을 상태로" ]                │        거절 + 사유 기록               │
  Operator ──권한 modifier 통과──▶ 온체인 상태 ────────▶│   ← 엔진은 이 상태를 "읽기만" 함       │
   (자격 인정·flag·예외 해제 같은 판단 결과)            │   (Operator 호출 코드 없음 ✋)        │
                                                   └──────────────────────────────────┘
        [ 사후 채널 ]
  엔진 ──거절 로그·의심거래 flag 큐──▶ Operator ──판단·처리·기록──▶ (보고·해제·신고)
```

**거래소 비유 (한국거래소 감각)**: 매매체결 엔진은 자동으로 돌고, 시장운영본부는 엔진 안에 손을 넣지 않습니다 — ① 상장 심사(사전) ② 거래정지 종목 *지정*(상태 입력 — 엔진이 읽고 자동 반영) ③ 시장감시(사후)로만 개입합니다. 우리 Operator가 정확히 이 위치입니다.

**왜 이렇게 하나**: 운영자가 체결에 재량 개입할 수 있으면 그 재량 자체가 이해상충·조작 의심·규제 리스크가 됩니다. 개입이 "서명된 상태 입력"으로만 가능하면, 감독·감사가 코드 전체를 뒤지는 대신 **입력 진입점의 권한과 서명 기록만 보면 되는 구조** — 감사 대상이 "과정"에서 "데이터"로 바뀝니다.

## 3. 네 가지 규칙 (왜 그런지까지)

1. **판단은 상태가 되어야 효력이 있다.** 운영자의 판단(예: "이 투자자는 자격이 있다", "이 거절 건은 예외에 해당한다")은 말로는 효력이 없고, **온체인 상태로 입력되어야** 작동합니다. 이때 게이트가 둘입니다 — *입력 게이트* (권한 있는 주소만 쓸 수 있음: 예를 들어 자격 명부는 지정된 검증기관만 수정 가능) + *참조 게이트* (엔진이 거래 시점에 그 상태를 기계적으로 읽음). ERC-3643이 실제로 이 2단 구조입니다: `onlyIdentityManager`(쓰기 통제) + `onlyValidInvestor`(읽기 참조).
2. **느슨하게 풀어주는 방향의 개입은 불가.** 자산별 설정 조정(override)은 *컴플라이언스를 강화하는 방향만* 허용됩니다. 반대 방향이 가능하면 운영자 만능 권한(God Mode)이 되므로 타입 차원에서 차단하고, 모든 조정은 법적 검토 단계를 거쳐 제안자·승인자가 기록됩니다.
3. **단일 키 금지 — 전부 다중 서명 + 시차.** 규칙 변경의 무게에 비례해 요건이 올라갑니다: 부품(Element) 단위 변경 = 2인/3인 서명 + 24시간 대기 / Recipe 단위 = 3인/5인 + 48시간 / 엔진 핵심 = 4인/7인 + 외부 감사 + 72시간. 시차(time-lock)는 "잘못된 변경을 발견하고 멈출 시간"입니다.
4. **남이 확인한 사실은 믿되, 자기 의무는 자기가 진다.** 발행 쪽이 검증한 사실(KYC·자격)은 재검증하지 않지만, 운영 주체 자신의 법적 의무 — 고객확인·의심거래보고·기록보존 — 와 그 기록은 운영 주체 명의로 남깁니다. *"재검증 안 함 ≠ 기록 안 함."*

## 4. 층별 상호작용 — 무엇을, 어떤 통제로, 무엇은 금지

### 4.1 Operator ↔ Element (검사 부품)

| 하는 것 | 구체적으로 | 통제 수단 |
|--------|----------|----------|
| 판단 결과 입력 | 증명서형·감시형 부품이 읽을 상태(자격 인정·flag 지정·해제)를 입력 | 입력 게이트 modifier + 서명 기록 |
| flag 처리 | 시장감시 부품이 올린 의심거래 flag를 큐로 받아 판단 → 처리 결과(승인·해제·신고 여부) 기록 | 처리 기록 필드 의무 |
| 예외 해제 | 차단된 거래가 법적 예외에 해당하면 해제 — 해제 자체도 "상태 입력"으로 | 거절 → 판단 → 해제 → 기록의 사슬 |
| 부품 버전업 심사 | 규제 변경 시 부품 개정안을 심사 | 2-of-3 서명 + 24h |
| ✋ **금지** | **카운터(보유자 수 등)에 직접 쓰기** — 카운터는 부품이 체결 후 자동으로만 갱신. 운영자가 숫자를 만지는 순간 그 숫자는 증거 가치를 잃음 | 구조적 차단 |

### 4.2 Operator ↔ Recipe (법률 묶음)

| 하는 것 | 구체적으로 | 통제 수단 |
|--------|----------|----------|
| 등록·버전업 심사 | 6단계 심사(신청→기술→**법적 검증**→운영→승인→모니터링). 운영자는 *심사자*이지 작성 독점자가 아님 — 제3자도 새 Recipe를 제안할 수 있음 ("열린 문에 문지기") | 3-of-5 서명 + 48h |
| ✋ **금지** | **거래 시점의 Recipe 내용 개입** — 런타임에서 Recipe는 불가침. 바꾸려면 버전업 절차뿐. "오늘만 이 요건 빼고 체결"은 구조적으로 불가능 | 구조적 차단 |

### 4.3 Operator ↔ Manifest (자산 신상카드)

| 하는 것 | 구체적으로 | 통제 수단 |
|--------|----------|----------|
| 온보딩 검증·등록 | 발행자가 제출한 신상카드 초안(적용 법률·명단·기준값)을 검증·조정해 온체인 등록 — 상장 심사에 해당 | 6단계 심사 + 발행자의 사실 진실성 보증 |
| 설정 조정 승인 | 자산별 부품 동작 조정 — **강화 방향만** (§3-2) | 법적 검토 + 기록 |
| 사실 갱신 | "판매 중 여부"·"매수 금지 명단" 같은 사실 변경 — 발행자 신고 → 운영자 승인 → 입력 | 입력 게이트 |
| 일괄 정비 | 부품 버전업 후 영향받는 모든 자산의 신상카드를 한 번에 갱신 | batch + 다중 서명 확인 |
| ✋ **금지** | 신상카드가 runtime 데이터(카운터)를 직접 쓰는 것 — 지휘자는 악보를 들지 연주하지 않음. 운영자도 동일 | 구조적 차단 |

### 4.4 Operator ↔ 매칭 엔진 (runtime)

| 하는 것 | 구체적으로 | 통제 수단 |
|--------|----------|----------|
| ✋ **대원칙** | **체결 경로에 Operator 호출 없음** — 엔진은 신상카드·Recipe·부품만 읽고 자율 실행 | 구조 자체 |
| 긴급 정지 | 특정 자산 또는 전체의 pause/freeze — 거래소의 거래정지와 같은 위상. 엔진 코드를 고치는 게 아니라 "정지 상태"를 입력하는 방식 | 다중 서명 + 사유 기록 + 해제 절차 (상세 규칙은 §6-1에서 함께 결정) |
| 기록 수신 | 거절 로그·판정 로그·카운터 이력을 받아 **규제 보존 양식으로 내보냄** — 회사의 기록보존·보고 의무 이행의 원료 | 로그 불변성 |
| 발행측 강제조치 수신 | 발행자(명의개서대행)의 강제이전·동결은 발행 쪽 권한 — 우리는 수신·정합 검사·기록만 | 신상카드 정합 검사 |

### 4.5 Operator ↔ 외부 (감독기관·발행자)

시스템이 자동으로 만들어 주는 것(자격 확인 사실·flag 이력·판정 로그·체결 규칙 명세) ↔ 회사가 직접 이행하는 것(고객확인·의심거래보고·자본 규제·인가 신청·최선집행 분석)의 분업. 인가 신청 서류의 기재 사항 상당 부분(체결 규칙·이용자 차별 여부·수수료 구조)이 **우리 아키텍처 문서의 내보내기**로 채워집니다 — 서류 부담이 설계 단계에서 이미 줄어 있는 구조.

## 5. 한 문단 요약

운영자는 엔진 밖에 서 있다. 사전에는 심사로, 거래 중에는 서명된 상태 입력으로, 사후에는 기록된 판단으로만 시스템에 개입한다. 그래서 차단은 코드가 하고, 사람은 코드가 못 하는 판단만 하며, 모든 개입은 데이터로 남는다 — 규제가 요구하는 내부통제를 구조 자체로 증명하는 방식이다.
