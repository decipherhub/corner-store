## 0. 이 문서가 하는 일 — 한 문단 요약

한국 감각으로 말하면 이 문서는 **"새 펀드 상품을 우리 거래소에 상장시킬 때 준비해야 할 것의 전체 목록"**입니다. 거래소가 새 상품을 받으려면 ① 상품의 신상정보를 등록하고(상장 심사 서류) ② 누가 살 수 있는 상품인지 자격 검사 체계를 연결하고(기관전용 상품이면 기관 확인) ③ 명의개서대행사와 명부를 연동하고 ④ 시장감시 규칙을 세팅합니다. 우리 시스템에서 그 각각이 무엇에 해당하고, 이번에 **새로 만들어야 하는 것이 무엇인지 — 결론: 5개뿐 —** 를 정리합니다.

> **전제 확인 (체인)**: 우리 base는 **Giwa 체인** (두나무 계열 Ethereum L2 — EVM 호환). ERC-3643은 이더리움 계열 표준이므로 **Giwa에 그대로 배포 가능** — 별도 호환 작업이 필요 없습니다 (정정 노트). 남는 인프라 질문은 하나: 현실의 BUIDL은 이더리움 등 6개 체인에 있으므로, *Giwa로 어떻게 가져오나* (다리 놓기 bridge vs 거울 토큰 mirror vs Giwa에서 새로 발행 — V-4). Giwa에서 새로 발행한다고 가정하면 이 질문도 사라집니다.

### BUIDL이 뭔가요 — 신상카드 먼저

**BlackRock이 만든 토큰화 MMF**입니다. 한국 감각: *기관전용 사모 MMF의 수익증권을 블록체인 토큰으로 발행한 것*. 기준가는 1달러 고정을 목표로 운용하고(미국 국채·RP·현금 편입), 이자는 매일 쌓아 매월 지급합니다. 판매와 명의개서는 Securitize가 맡습니다 (미국 SEC 등록 transfer agent — 한국의 **명의개서대행회사 + 펀드 판매대행**을 합친 감각).

| 항목 | 내용 | 우리에게 갖는 의미 |
|------|------|------------------|
| 법적 형태 | 사모 펀드 (ICA §3(c)(7) 면제 + Reg D 506(c) 발행) | 공모펀드 규제를 면제받는 대신 **qualified purchaser(QP — 대형 기관·거액 자산가) 한정** |
| 최소 투자 | **$5M, QP만** | 매수인마다 "QP 맞는지" 검사 필요 — 적격투자자(accredited investor)보다 **한 단계 높은** 기준 |
| 기준가(NAV) | $1 고정 목표 | "시세 받치기" 우려가 구조적으로 없음 — 비조치의견서 논거 (유통규제 10) |
| 수익 분배 | 매일 적립 → **매월 지급** (신규 토큰 지급 방식 — 일부 자료는 rebase로 기술, 확인 항목 V-1) | 매월 지급분 = **신규 발행** → 발행 검사를 거쳐야 함 (§IV.5) |

| 발행 방식 | **매일 상시 발행** (continuous offering) | "판매 기간이 끝나지 않는다" → Reg M 매수 금지가 상시 작동 (유통규제 10) |

## I. 가정의 정리 — "ERC-3643으로 구현되었다"는 게 무슨 뜻인가

현실의 BUIDL은 Securitize의 자체 방식으로 만든 허가형 토큰입니다. 본 시나리오는 회의 기획대로 **BUIDL이 ERC-3643이라는 표준 부품 세트로 재구성되어 있다**고 가정합니다. ERC-3643 부품들을 한국 증권 인프라에 비유하면:

| ERC-3643 부품 | 비유 (한국 인프라) | 하는 일 |
|--------------|------------------|--------|
| **Token** | 전자증권 그 자체 | 보통 토큰(ERC-20)에 "이전할 때마다 자격·규칙 검사"가 붙은 것 |
| **ONCHAINID** | 투자자별 **전자 신분증 지갑** | 한 투자자의 각종 자격 증명서를 담아두는 온체인 보관함 |
| **Claim (자격 증명)** | 공인기관 발급 **확인서** | "이 사람 KYC 통과" "이 사람 QP 맞음" — 검증기관이 서명한 디지털 확인서 |
| **Trusted Issuers Registry** | **공인 검증기관 명단** | 어느 기관이 발급한 확인서를 믿을지의 목록 (여기선 Securitize) |
| **Identity Registry** | **주주명부 + 적격자 명부** | "이 지갑 = 검증된 투자자"의 공식 매핑 — 화이트리스트의 표준화 형태 |
| **Modular Compliance** | **이전 시 자동 검사 규칙집** | 발행자가 정한 규칙 (국가 제한·보유자 수 상한 등)을 이전 순간 자동 검사 |
| **Agent 권한** | **명의개서대행의 강제 권한** | 강제이전·동결·계좌복구·발행·소각 — 법원 명령·상속 처리 같은 상황용 권한 |

→ 쉽게 말해 ERC-3643은 *"전자증권법 + 명의개서대행 + 적격투자자 명부를 코드로 만든 표준"*이고, BUIDL이 이 표준으로 존재한다고 가정하는 것입니다.

## II. 분업의 대원칙 — 발행 쪽이 이미 한 일은 다시 안 한다

은행 비유: 고객이 *주민센터에서 발급받은 신분증명서*를 들고 오면 은행은 그 서류를 믿지, 주민센터 일을 다시 하지 않습니다. 대신 *은행만 아는 것* (이 고객의 계좌 이력, 오늘 거래의 맥락)은 은행이 직접 봅니다. 우리 설계의 **Coverage Delta** 원칙이 정확히 이것입니다 (09):

| 검사 항목                     | 누가 하나                                          | 이유                               |
| ------------------------- | ---------------------------------------------- | -------------------------------- |
| 매수인 자격 (QP·KYC·제재 명단)     | **발행 쪽** (Securitize 확인서) — 우리는 그 확인서를 *믿고 소비* | 발행 쪽이 이미 했고, 잘하는 일               |
| 매도인 쪽 사실 (얼마나 보유했나·내부자인가) | **우리**                                         | 발행 쪽은 *2차 거래의 매도인*을 구조적으로 못 봄    |
| 거래 맥락 (방향·수량·시장 행위 패턴)    | **우리**                                         | 거래소만 볼 수 있는 영역                   |
| 발행자 규칙 (보유자 수 상한 등)       | 발행 쪽 규칙집이 이전 순간 **자체 검사**                      | 우리 검사와 *분담* — 같은 걸 두 번 검사하는 게 아님 |

> ⚠️ **한정 — 사실은 공유해도, 의무는 각자의 것** (09 조건 5 한정·G2, 2026-06-08 사용자 지적으로 확립): Coverage Delta는 *사실(facts)의 중복 검증 제거*이지 *의무(duties)의 면제*가 아닙니다. 은행 비유를 정확히 하면 — 은행은 주민센터 증명서를 믿고 *신원 확인 작업*을 반복하지 않지만, **고객확인의무(CDD)·의심거래보고(STR)라는 은행 자신의 법적 의무와 그 기록은 여전히 은행 명의로 이행**합니다. 우리도 같습니다: Securitize의 QP·KYC 확인서를 믿고 소비하더라도, **운영주체(BD)의 CIP(미국판 고객확인)·SAR(의심거래보고)·기록보존 의무는 소멸하지 않으며**, "무엇에 의존했는지"를 포함한 이행 기록을 자기 명의로 남깁니다 — **"재검증 안 함 ≠ 기록 안 함"**. Securitize는 금융기관 간 정식 reliance가 아닌 agent/외주 구조일 수 있어 그 적합성은 변호사 위임 중 (Q-O1).

## III. 구현 목록 ① — BUIDL 신상카드 작성 (가장 먼저, 코드 아닌 데이터 작업)

우리 시스템에서 자산마다 붙는 신상카드(Manifest/ACM — 07)의 **BUIDL 버전을 작성하는 것이 첫 산출물**입니다. 상장 심사 서류와 같은 위상입니다. 채울 값:

```yaml
# BUIDL 신상카드 (핵심 칸만 — 한국어 주석)
asset: BUIDL
tokenStandard: ERC-3643 (가정)
fundForm: ICA-3c7            # "기관전용 사모펀드"라는 선언 → QP 검사(A-13)가 자동으로 켜짐
issuanceExemption: RegD-506c # 발행 면제 근거 → 해당 검사 묶음(Recipe) 연결
reportingStatus: non-reporting  # 미국판 사업보고서 제출 대상 아님
nmsStatus: non-NMS              # 상장 대형주 아님 (ATS 신고서식 선택의 근거)
distributionStatus:             # ⭐ "지금 판매 중인가" — Reg M 검사용
  active: true                  # 상시 발행이므로 항상 true
  basis: CONSERVATIVE_DEFAULT   # 보수적 기본값. 비조치의견서 받으면 이 값만 바꿈 (코드 무변경)
  restrictedParties:            # "매수 금지 명단" = 파는 쪽 사람들
    - BlackRock 및 계열 (affiliated purchasers)
    - Securitize (판매 참여 entity)
12gThresholds: { recordHolders: 2000, nonAccredited: 500 }  # 보유자 수 경보 기준 (미국판 공시의무 진입 경계)
navSource: { type: oracle, target: $1.00 }                  # 기준가 정보 연결
redemptionRail: { usd: 영업일, usdc: 즉시(Circle) }          # 환매 창구 식별 (DEX와 분리 대상)
```

**이 카드 한 장으로 결정되는 것**: QP 검사 켜짐, Reg M 검사 상시 작동, 보유자 수 경보 기준. **자산 온보딩의 절반은 이 카드 작성**입니다 — 코드는 그대로고 데이터만 바뀝니다 (명제 1).

## IV. 구현 목록 ② — 적용되는 검사들 + 새로 만드는 것 5개

### IV.1 한 거래에 적용되는 법률 — 전체 스택 (증권법 4겹 + 횡단 규제)

김 부장(QP인 기관 A)이 다른 기관 B에게 BUIDL을 파는 한 번의 거래에 적용되는 법률의 **전체 지도**입니다. 두 부류로 나뉩니다: **(a) 증권법 묶음 4겹** — 자산·경로에 따라 조합이 달라지는 Recipe들 / **(b) 횡단 규제** — 증권법과 *무관하게, 어떤 거래에든 독립 적용*되는 별개 법률들 (제재법·AML·세법·주법). 자본시장법 감각으로: 한 거래에 발행공시 + 전매 제한 + 사모펀드 규제 + 불공정거래가 따라붙고, **그와 별도로 특금법·외국환거래법·세법이 또 따라붙는 것**과 같습니다.

**(a) 증권법 묶음 4겹 (Recipe — 자산·경로 따라 조합)**

| 겹                   | 무엇을 검사                               | 한국법 감각              |
| ------------------- | ------------------------------------ | ------------------- |
| ① 발행 (Reg D 506(c)) | 매수인이 발행 면제의 자격 요건(적격투자자 등)을 갖췄나      | 사모 발행 요건            |
| ② 전매 (Resale)       | 매도인이 팔아도 되는 상태인가 (보유기간·내부자 여부·면제 경로) | 전매 제한 — 보호예수 감각     |
| ③ 펀드 (ICA 3(c)(7))  | **매수인이 QP인가** ⭐ BUIDL의 핵심 추가 검사      | 기관전용 사모펀드의 투자자 제한   |
| ④ 시장행위 (상시)         | 시장감시 + **Reg M — 파는 쪽 사람의 매수 금지**    | 제4편 불공정거래 + 안정조작 금지 |

**(b) 횡단 규제 (증권법 아님 — 독립 법률, 모든 거래에 적용)**

| 법률 영역 | 근거법 | 무엇을 요구 | 우리 시스템 내 위치 | BUIDL 구현 상태 |
|----------|--------|-----------|-------------------|----------------|
| **제재 (Sanctions)** | OFAC·IEEPA (증권법 아닌 국가안보법) | 제재 대상자와는 *어떤 거래도* 금지 — strict liability (모르고 해도 위반) | A-01 검사 부품 — **상시 (always-on)** | ✅ Phase 2부터 |
| **AML/KYC** | BSA (은행비밀법 — 특금법 대응) | 고객확인(CIP)·의심거래보고(SAR)·기록 — *운영주체(BD)의 의무* | 사실은 검사 부품이 공급 (A-01·A-04) + **의무는 Operator plane** (§II 한정) | ✅ 사실 / 의무는 BD 등록 후 |
| **관할·역외** | 각국 증권법·역외 규정 (Morrison) | 허용 관할 투자자만 (Phase 1·2 non-US 등) | A-02 검사 부품 — **상시** | ✅ Phase 2부터 |
| **세법** | IRS (FATCA·정보 보고) | 미국 거주자 거래 시 보고 의무 리스크 | A-05 (운영상 리스크 차단) + Operator plane | 🟡 Phase 2 optional |
| **주법 (Blue Sky)** | 50개 주 증권법 | Reg D여도 주별 notice filing (연방-주 이중 구조 — 한국에 없음) | G-02 추적 부품 — L5 | 🟡 Phase 3 (US 진출 시) |
| (경계 확인) 상품·파생 | CFTC 관할 | BUIDL은 증권이므로 비해당 — *경계 확인만* | 신상카드 facts | — 해당 없음 기록 |

> **⚠️ 구조 노트 (사용자 지적 2026-06-12)**: 횡단 규제 중 제재(A-01)·관할(A-02)은 현재 Element pool 분류상 *Reg D Recipe(L1) 소속*이지만, **법적 성격은 증권법 면제 요건이 아니라 독립 법률** — 시장행위(④)와 같은 **always-on**으로 재분류하는 것이 "1 법률효과 = 1 Recipe" 원칙에 더 정확하다. 검사 실행은 동일하므로 *코드 변경이 아니라 분류 변경* — Element pool freeze 전 결정 항목 (**OD-CI-5**, 15 §V). **구현은 지금처럼 하더라도, 자산별 논증서(12)의 법률 지도는 (a)+(b) 전체를 보여야 "규제를 다 지켰다"는 논증이 완성된다.**

### IV.2 쓰이는 부품 수

기존 검사 부품(Element) 41개 중 **27개가 BUIDL에서 활성화**됩니다 — 현재 pool에서 활성률이 가장 높은 자산. *그래서 BUIDL이 첫 시연 자산으로 적합*합니다 (한 자산으로 시스템 전체를 검증).

> **횡단 규제를 반영해도 이 수는 변하지 않습니다** (2026-06-12 확인): 제재(A-01)·관할(A-02)·신원(A-04)은 *이미 27개 안에 포함*되어 있고, §IV.1(b)의 재분류(OD-CI-5)는 "어느 묶음 소속인가"를 바꿀 뿐 부품을 추가·제거하지 않습니다. 단 세법(A-05)·Blue Sky(G-02)는 *원래 Phase 후순위라 27에 미포함* — Phase 2에서 켜기로 결정하면 그때 +2 (27→29)가 됩니다.

### IV.4 거래 한 건 걸어보기 (walkthrough)

```
김 부장(기관 A, QP) → 기관 B에게 BUIDL 매도 주문

1. 주문 접수 (DEX 2차 시장 통로)
2. 신상카드 로드 → "이 자산엔 4겹 검사 + 이런 사실들" 확인
3. 검사 일괄 실행:
   매수인 B 쪽:  제재명단 ✓ / 허용 관할 ✓ / 적격투자자+QP 확인서(Securitize 발급) ✓ / 확인서 유효기간 ✓
   매도인 A 쪽:  전매 면제 경로 선택 ✓ / 보유기간 ✓ / 내부자(affiliate) 아님 ✓
   거래 자체:    A·B 모두 "매수 금지 명단" 아님 ✓ (F-04) / 시장행위 이상 패턴 없음 ✓
4. 전부 통과 → 단일 트랜잭션으로 체결 (Giwa)
   하나라도 걸림 → 거절 + 사유 코드 기록 (이 기록이 곧 미국 기록유지 규정 17a-3/4 대응 자산)
5. 사후: 보유자 수 카운터 갱신 / 시장감시 이력 적재
```

참고: **발행 쪽 규칙집(Modular Compliance)도 토큰 이전 순간 자기 검사를 따로 돌립니다** — 같은 걸 두 번 검사하는 게 아니라 분담입니다 (§II).

### IV.5 발행자 쪽 행위가 들어올 때 — 특수 통로 3개

| 상황 | 처리 |
|------|------|
| 매월 이자 지급 (신규 토큰) | N-4 간이 통로 (1차 시장) |
| 투자자 환매 (USD·USDC) | 1차 시장 통로 — F-04 검사 비적용 구간 (분리 자체가 방어) |
