# 쟁점 후보 등록부 (Open Design Issues — Candidate Register)

> **한 줄:** 볼트를 전수 스캔해 긁어낸, *아직 안 닫힌 설계·법률 쟁점*의 후보 목록. 기억으로 적은 게 아니라 스캔으로 떠올린 것이라 누락이 적다.

---

## 0. 이게 뭐고, 어떻게 쓰나

**쉽게 말하면**, 이건 "처리 안 된 숙제들"을 한 장에 모은 칠판이다. 이걸 보고 변호사님이 할 일은 두 가지 — ① 어떤 숙제를 *지금* 토론용 제안서(RFC)로 띄울지 **고르고(triage)**, ② 그걸 *어디서* 토론할지(GitHub vs Notion) **정하는** 것.

이 표는 *결정을 내리는 곳*이 아니다. 결정이 나면 그건 별도의 "의사결정 등록부"(다음 PM 문서)로 옮겨 닫는다. 여기는 *"열려 있다"를 빠뜨리지 않고 보이게* 하는 게 일이다.

**상태 범례**
- 🔴 지금 freeze(부품 목록 확정)를 막는 것 — 최우선
- 🟡 freeze 후 / Phase 1~2 중 결정
- ⚪ 후순위·보류 (Phase 3+ 또는 외부 의존)
- ✅ 최근 닫힘 (다시 안 열도록 참고용)

**토론 채널 범례** (주제별 분리 — 사용자 06-13 결정)
- **[G]** GitHub Issue/ADR — 코드에 직접 연결되는 기술 결정. 개발팀이 거기 삶
- **[N]** Notion — 법률 substantive·운영 정책. 보경 변호사·승환 등 비개발자 토론
- **[둘]** 양쪽 — 기술 결정이되 법률 판단이 섞임

## 6. 트리아지 안내 — 어떻게 고르나

1. **1순위는 §1(freeze 막는 9건).** 노트 31의 "Sprint 1 결정 세션"에서 한자리에 닫으면, Phase 1이 열린다. 닫힌 항목은 의사결정 등록부(다음 PM 문서)로 이관.
2. **RFC/ADR 첫 시범 추천:** §2의 *3축→5축* 또는 §1의 *OD-CI-5*를 GitHub ADR 한 장(배경·옵션·권고·열린 질문)으로 만들어, "팀이 토론 가능한 형태"의 *첫 사례*를 만든다. 한 번 형식이 잡히면 나머지는 같은 틀로 찍어낸다.
3. **채널:** [G]는 GitHub Issue/ADR, [N]은 Notion. [둘]은 기술 spec은 GitHub, 정책 합의는 Notion에 두고 서로 링크.
