# 프로젝트 진행 연대기 + 구현 맵

> **한 줄:** 우리가 04-23부터 지금까지 어떻게 생각을 발전시켜 왔고(연대기), 그 목표가 코드로 어디까지 구현됐으며(구현 맵), 지금 무엇이 막혀 있는지(병목)를 한 장에 모은 PM 진입 문서.

---

## 0. 이 문서가 뭐고, 왜 첫 번째인가

**쉽게 말하면**, 이건 등산 지도의 "현재 위치" 핀이다. 프로젝트 폴더에 흩어진 30여 개 분석 노트가 등산로 곳곳의 표지판이라면, 이 문서는 그걸 한 장에 모아 "지나온 길 + 지금 위치 + 가야 할 곳"을 보여준다.

읽는 사람은 둘이다. 첫째는 PM인 나 자신 — 판단을 내릴 때마다 30개 노트를 다시 읽어 상태를 재구성하지 않으려고. 둘째는 새로 합류하는 팀원이나 위임 변호사 — 5분 안에 프로젝트를 따라잡으려고.

**왜 "블로커(열린 결정 목록)"보다 이 연대기가 먼저인가.** 결정 목록은 *"지금 열려 있는 것"*만 평평하게 보여준다. 하지만 그 결정 하나하나가 *무엇을 향한 결정인지*는 거기서 안 보인다. 법률 메모로 치면, 결정 목록은 "쟁점 목록"이고 이 연대기는 그 앞에 오는 **"사실관계·경과(procedural history)" 절**이다. 경과를 먼저 깔아야 각 쟁점이 어디서 왜 생겼는지가 잡힌다.

**원본은 볼트, GitHub는 종착점.** 이 프로젝트는 내 볼트 분석이 먼저 나왔고, 코드(GitHub `decipherhub/corner-store`)가 나중에 그걸 받아 구현으로 굳혔다. 그래서 사고 발전의 줄기는 볼트 노트(특히 회의록)에서 뽑고, 코드는 "지금까지 어디까지 반영됐나"의 *종착점*으로만 참조한다.

---

## 1. 우리가 만드는 것 (한 문단)

**정체성 한 문장** (목표·원칙 종합본, 노트 25): *"ERC-3643이 token transfer eligibility(토큰을 누가 받을 수 있나)의 표준이듯, 우리는 market access eligibility(이 시장에 누가 어떻게 참여할 수 있나)의 표준이다."*

**쉽게 말하면**, 일반 토큰 거래소(유니스왑 같은 곳)는 누구나 아무 토큰이나 사고판다. 우리는 증권형 토큰(실물자산을 토큰화한 RWA)을 대상으로, *"이 사람이, 이 증권 토큰을, 이 경로로, 이 수량만큼 거래해도 적법한가?"*를 거래가 체결되기 **직전에 코드가 자동으로 판정**하게 만든다. **한국법 감각으로 옮기면**, 자본시장법의 적격투자자·전매제한·시장질서 규정을 사람이 일일이 확인하는 대신, 스마트컨트랙트가 자동 게이트로 거는 것이다.

만드는 것의 주종 순서는 이렇다.

| 순위 | 무엇 | 쉽게 말하면 |
|------|------|-------------|
| 주력 제품 | DEX-level 컴플라이언스 표준 + SDK | 누구든 가져다 쓸 수 있는 "거래소용 준법 부품 라이브러리" |
| 레퍼런스 구현 | Corner Store DEX (코드네임 구멍가게) | 그 표준이 실제로 돌아감을 증명하는 데모 거래소 |
| 1차 적용 자산 | 미국 Reg D 506(c) RWA의 2차 유통 | 블랙록 BUIDL·온도 USDY 같은 토큰화 펀드 |

체인은 Giwa(두나무 계열 OP Stack EVM L2), 타깃 시장은 미국(SEC의 Reg D 506(c) 사모 + Reg ATS 대체거래시스템 경로)이다.

---

## 2. 두 개의 세계, 그리고 내 자리

이 프로젝트엔 문서가 사는 세계가 둘 있다.

- **세계 A — 법률 분석 (내 옵시디언 볼트, 노트 100여 개).** "어떤 법이 무엇을 요구하는가."
- **세계 B — 실제 구현 (GitHub `corner-store`, Solidity 코드).** "그 요구를 코드로 어떻게 짓는가."

**내 역할은 두 세계의 다리다 (리걸 카운슬 겸 PM).** 건축에 빗대면 나는 "법규 검토 + 현장소장"을 겸한다 — 설계사(법)와 시공팀(코드) 사이의 통역이자, 공정 순서를 관리하는 사람.

⚠️ **그리고 여기에 병목의 근원이 있다.** 지금 두 세계를 잇는 *공식 문서가 없다.* 코드 쪽 로드맵을 보면 개발팀이 착수 조건으로 **"Legal-to-Technical Matrix"(법률요건 → 코드 변환표)**를 콕 집어 기다리고 있는데(near-term 이슈 #4), 이게 미작성이다. **즉 개발이 못 나가는 이유가 내 산출물 미제출** — 병목이 PM/리걸인 나에게 있다는 게 코드 쪽에서도 똑같이 확인된다.

## 3. 생각이 발전해 온 길 (시간순)

각 분기점마다 *무엇을 이해·결정했고, 무엇이 새로 열렸는지*를 적는다.

### 2026-04-23 — 출발: "한국에선 안 된다"

저유동성 토큰화 증권을 위한 **허가형 RFQ DEX**(요청-견적 방식 거래소) 구상에서 출발. 초기 법률검토에서 한국 자본시장법·전자증권법상 운영 불가를 조문 단위로 확인하고, 미국·EU·싱가포르를 비교해 **미국의 브로커-딜러·ATS 정통 경로**를 1순위로 권고했다. 증권형 자산이므로 SEC가 1차 규제기관(CFTC·Blue Sky는 보조).

- 열린 것: 어느 라이선스 경로로, 컴플라이언스를 어떻게 코드로 구조화할 것인가?

### 2026-05-20 — 아키텍처 골격: ERC-3643 + 3-Layer

팀 회의에서 **ERC-3643(증권형 토큰 표준) 호환 RWA DEX + 3-Layer 아키텍처**에 합의. 핵심 결정 네 가지가 이때 섰다. ① 일반 가상자산 DEX를 데모로 "갖다붙이는" 안은 컨셉 불일치로 접음. ② AMM(자동 시장조성) 일변도에서 벗어나 미국 규제 적합성이 높은 온체인 오더북도 검토. ③ ERC-3643 네이티브 발행 자산만 취급(브릿지 자산은 transfer 통제 불가). ④ MVP의 핵심은 "3-Layer가 실제로 돌아간다는 것의 시연"으로 정의. 이 시점은 가속 프로그램(Gasok) 5/31 지원 준비기였다.

- 열린 것: 첫 자산군은? 운영주체(법적 주체)는 누가?

### 2026-05-27 — ⭐ SDK 피벗 (가장 큰 방향 전환)

프로젝트 포지셔닝을 **"RWA-Compliant DEX"에서 "RWA Compliance SDK + 데모 DEX"로 전환**. 이게 전체 narrative를 바꾼 분기점이다.

**쉽게 말하면**, "우리가 거래소를 운영하겠다"가 아니라 "누구든 우리 부품으로 준법 거래소를 만들 수 있게 도구를 제공하겠다"로 바뀐 것. 이 전환이 중요한 이유는, *"법적 주체(운영주체)가 없으면 거래소가 작동할 수 없다"*는 구조적 난제를 한 번에 우회하기 때문이다 — SDK는 인프라(도구)이지 운영 주체가 아니므로, 운영주체 리스크가 SDK *사용자*에게 외주화된다. 동시에 발행 측 컴플라이언스(ERC-3643이 담당)와 거래 측 컴플라이언스(우리 Element·Recipe가 담당)가 명확히 분리됐다.

- 열린 것: SDK 제공자는 법적으로 무엇인가(브로커-딜러? UI Provider 면제 대상?)?

### 2026-06-03 — post-Gasok 전환 + 다국가 통찰

가속 지원을 마친 뒤 첫 회의. 두 가지가 정해졌다.

첫째, **구현은 step-by-step 빌드업으로** — 가장 단순한 AMM 테스트부터 짓고, 작동하면 RWA·Reg ATS 단계로 확대. *"개발에서 병목이 어디서 날지 모른다"*는 인식이 근거였다. 그리고 **각 시나리오를 정의하는 책임이 나(리걸)에게** 배정됐다(개발팀이 그걸 받아 구현). 체인은 public·EVM 호환만, Canton은 보류.

둘째이자 더 중요한, **⭐ Element 다국가 contextuality 통찰** — 내가 제기했다. *같은 사실관계도 관할(미국·EU·한국·싱가포르)마다 다르게 평가된다.* 예컨대 "적격투자자"의 정의·임계치가 나라마다 다르다. 따라서 Element(검증 부품)는 단순 참/거짓이 아니라 *맥락(jurisdiction)을 입력받아야* 한다는 인식. 이건 "Element는 어디서나 재사용 가능하다"는 기존 가정에 대한 정면 도전이었다.

- 열린 것: 관할 × Element 해석 매트릭스, 시연 시나리오 7종.

### 2026-06-08 — 목표·원칙·구현의 종합 (현재의 기준틀)

흩어진 방향을 **목표 → 설계 원칙 6개(G1~G6) → 4-Layer 구현**의 한 페이지로 종합(노트 25). 정체성(§1의 한 문장)이 이때 확정됐다. 6개 원칙을 쉽게 풀면: 위반 거래는 아예 체결 안 됨(G1), 이미 검증된 사실은 다시 안 봄(G2), 새 규제는 코드 수정이 아니라 데이터 등록으로 흡수(G3), 무겁거나 못 푸는 건 오프체인·온체인은 검증만(G4), 책임은 각자에게 기록으로 귀속(G5), 쓸수록 싸짐(G6).

### 2026-06-09 ~ 06-12 — 심화와 구체화 (지금 위치)

4-Layer를 상세 spec으로 내려쓰고(Element·Recipe·Manifest·Operator 아키텍처 노트들), 기존 플랫폼(Securitize·tZERO) 대비 차별점을 "Protocol Portability"(어느 규제환경에서도 같은 프로토콜 재사용)로 정리했다. **블랙록 BUIDL 펀드를 ERC-3643으로 구현한다면?**을 첫 구체 인스턴스로 풀었고(노트 29), 유통규제 Reg M(공모 중 시세 받치기 금지)을 심화했으며, **실행 플랜(노트 31)**과 **리걸 파트 구현 플랜(노트 33)**을 세웠다. 체인 베이스를 Canton에서 Giwa로 정정했다. 그 결과 **Element pool(부품 목록)을 확정(freeze)할 수 있는 상태**에 도달했다.

- 열린 것 = **지금의 병목**: Element pool freeze 결정 + Legal-to-Technical Matrix 작성 (→ §5·§6).

## 4. 4-Layer 아키텍처가 뭔지 (한 번에 잡기)

연대기 곳곳에 나오는 "Element·Recipe·Manifest·Operator"를 한 번에 정리한다. **쉽게 말하면 레고다.** 법을 작은 블록으로 쪼개고(Element), 블록을 조립해 하나의 규제를 만들고(Recipe), 어떤 토큰에 어떤 조립을 쓸지 선언하고(Manifest), 사람이 판단·감시하는 영역(Operator)을 둔다. 이 전체가 ERC-3643(발행 측 표준) 위에 얹힌다.

| Layer | 쉽게 말하면 | 법률 비유 (한국법 감각) | 코드(repo)에서의 이름 |
|-------|-------------|------------------------|----------------------|
| Element | 검증 부품 하나 | 규제의 "구성요건" 하나 (예: 적격투자자 여부) | ElementRegistry |
| Recipe | 부품의 조합 = 하나의 규제 | 하나의 규제 전체 (예: Reg D 506(c)) = 구성요건들의 묶음 | RecipeRegistry |
| Manifest | 이 토큰엔 이 Recipe 적용, 선언·기록 | 발행인의 신고 + 거래소의 승인 기록 | TokenPolicyRegistry |
| Operator | 사람의 판단·승인·감시 | 거래소 시장감시본부 / 운영주체 | OperatorRegistry |

핵심: Element는 *재사용*되도록 안정시키고, 새 규제·새 나라·새 자산은 Recipe·Manifest를 *등록*하는 것으로 흡수한다(원칙 G3). 그래서 "코드는 수정에 닫히고, 등록에 열린다."

## 5. 목표가 코드로 구현되는 단계 (구현 맵)

코드 쪽 로드맵은 Phase 0~6으로 나뉜다. 각 단계가 *내 어떤 법률 산출물을 먹고*, 지금 어디까지 왔는지를 맞붙인 표다. **이게 "목표가 차근차근 구현되고 있는지"를 확인하는 핵심 표다.**

| Phase | 코드가 하는 일 (쉽게 말하면) | 내 법률작업이 먹이는 것 | 상태 | 막는 것 |
|-------|------------------------------|------------------------|------|---------|
| 0 Foundation | 빈 골조·공통 부품·테스트 틀 깔기 | (법률 입력 불요) | ✅ 완료 | — |
| 1 Compliance Policy ⭐ | Element·Recipe·판정 엔진 구현 | Element pool 확정 + Legal-to-Technical Matrix | 🔴 병목 | 부품 확정 결정 미결 + 변환표 미작성 |
| 2 Execution & Routing | 허용된 거래만 거래소로 보내는 라우터 | 적음 (라우터는 법 판단 안 함) | ⬜ 대기 | Phase 1 |
| 3 Uniswap AMM (첫 venue, E2E) | 실제 거래소 연결 + BUIDL 시나리오 시연 | 시연 시나리오 7종 (06-03 합의, 내 책임) | ⬜ 대기 | Phase 1~2 + 수수료·풀 등록 합의 |
| 4 RFQ / 5 Order Book | 기관·지정가 거래 경로 | dealer·custody·matching 모델 자문 | ⬜ 후속 | 해당 모델 결정 |
| 6 Operations | 실제 출시·운영 | production 법률 출시 조건 | ⬜ 최후 | 거버넌스·출시 결정 |

**쉽게 말하면**, 시공팀은 1층 골조(Phase 0)까지 올렸다. 2층(Phase 1, 컴플라이언스 엔진)을 올리려면 내가 *"어떤 법적 부품을 쓸지 확정한 자재 명세서"*(Element freeze + Legal-to-Technical Matrix)를 줘야 한다. 그게 지금 안 나와서 공사가 멈춰 있다. 반복하지만, **병목은 PM/리걸인 나다.**

(참고: 볼트의 4-Layer 중 Manifest는 코드에선 TokenPolicyRegistry로, Operator는 OperatorRegistry로 landed. 즉 내 법률 설계가 코드 구조에 거의 1:1로 반영돼 있다 — 다리가 이미 절반은 놓여 있다는 뜻.)

## 6. 지금의 프론티어 = 병목 한 장

**막힌 지점은 Phase 1이다.** 이걸 열려면 네 가지가 필요하다.

1. **Element pool freeze 결정** — 부품 목록을 확정하는 결정들(내부 식별자 OD-CI-1~5, OD-B1~2로 추적 중). 큰 줄기는 ① Element를 어떻게 분류·재배치할지 ② 횡단 규제(제재·관할·Reg M)를 Element로 둘지 Recipe로 둘지 ③ BUIDL의 분배 방식 같은 구체 쟁점이다.
2. **Legal-to-Technical Matrix 작성** — 코드가 직접 요구하는, 법률요건 → 증거 → Element → 집행 지점 → 실패 시 동작을 잇는 변환표. 내 핵심 미제출 산출물.
3. **시연 시나리오 7종 정의** — 06-03에 내가 맡기로 한 작업(개발팀 hand-off용).
4. **위임 변호사 질문 정리** — 일부는 비블로킹(우회 가능).

**누가 누구를 기다리나** (PM이 매일 봐야 하는 한 줄):

- 개발팀 → **나**(Element freeze + Matrix)
- 보경 변호사 → **나의 검토 완료** (대량 자료 전달 후 대기 모드)
- 두나무 컨택 → 담당자 출장 복귀

**다음 액션** (실행 플랜 노트 31이 정의): Sprint 1 결정 세션 ①② (내가 1~2시간 들여 위 결정들을 닫는 자리) → freeze 선언 → 개발팀 공유.

→ 위 1~4의 세부 항목이 바로 **다음 PM 문서(블로커/의사결정 등록부)가 담을 행들**이다. 이 연대기가 "왜 이 결정들이 여기 있나"를 깔아줬으니, 다음 문서는 "그 결정들을 닫는 도구"가 된다.
