# PM 운영 규약 — 내가 이 프로젝트를 어떻게 운영하는가

> **한 줄:** 리걸 카운슬 겸 PM으로서 이 프로젝트를 운영하는 방식(역할·문서 체계·공유 체계·결정 프로세스·리듬)을 한 곳에 규정한다. 다른 PM 문서들이 "내용"이라면, 이 문서는 "운영 규칙"이다.

---

## 0. 이 문서의 자리

**쉽게 말하면**, 회사로 치면 이건 내 *직무기술서 + 업무 매뉴얼*이다. 연대기·쟁점 등록부·의사결정 등록부가 "무슨 일이 있었나"를 담는다면, 이 규약은 *"나는 어떻게 일하는가"*의 규칙이다. 운영 방식이 바뀌면 이 문서만 고친다(나머지는 이걸 참조).

처음 PM을 맡았고 이 역할이 익숙하지 않으므로, 규칙을 머릿속이 아니라 글로 박아둔다. 그래야 바쁠 때도 빠뜨리지 않는다.

---

## 1. 내 역할 — 리걸 카운슬 겸 PM

나는 모자를 두 개 쓴다.

- **리걸 카운슬:** 법을 *코드가 먹을 수 있는 사양(spec)*으로 번역하고, 법적 판단·자문을 제공한다.
- **PM:** 작업 순서·결정·병목·소통을 관리한다. *올바른 일이 올바른 순서로, 아무것도 빠지지 않고, 모두가 자기가 뭘 기다리는지 아는 상태*를 만든다.

**안 하는 일:** 직접 스마트컨트랙트 개발(개발팀 영역), 다른 팀원 작업의 세부 간섭.

**핵심 책임 한 줄:** *"법률 분석이 코드로 흐르게 하고, 팀이 같은 그림을 보게 한다."*

⚠️ **자각:** 지금 프로젝트의 병목은 나에게 있다(연대기 §6 — Element freeze·Matrix 미제출). 이 규약은 그 병목을 푸는 *일하는 방식*을 고정하기 위한 것이다.

## 2. 두 개의 surface — 무엇을 어디에 두나

> **2026-06-13 결정:** 공유 surface를 **GitHub로 일원화**(셋 → 둘). Notion 분리 폐기.

문서가 사는 곳은 둘이다.

| Surface | 역할 (쉽게 말하면) | 무엇의 SoT |
|---------|--------------------|------------|
| Obsidian 볼트 | 내 사고 연구실 — 깊은 법률분석 + 공유 문서 *초안 작성처* (비공유) | 법률 추론의 원본 |
| GitHub (`decipherhub/corner-store`) | 공유 정본 — 팀이 보는 모든 것 + 자동화 | spec·결정·쟁점·상태·오리엔테이션 전부 |

흐름: **Obsidian(사고·초안) → GitHub(공유 정본)**. 볼트에서 익은 것을 GitHub로 정제·발행하고, *발행된 공유 문서의 SoT는 GitHub*(편집은 PR로). 같은 내용을 두 곳에서 따로 관리하지 않는다.

**왜 GitHub 일원화인가** (결정 근거):
- **안전** — 모든 변경이 git 이력에 남고, 브랜치+PR로 작업해 실수해도 `revert`로 복구. (Notion 공유 워크스페이스 직접 편집은 blast radius가 크고 복구 어려움)
- **감사추적** — 결정마다 커밋·diff·작성자·시각 → 리걸 카운슬에게 법률 등급 기록.
- **자동화** — MCP/CI로 안정적 처리. (Notion MCP보다 안정적)

**Notion은?** Decipher 조직 가시성이 필요하면, Decipher Notion에 *"프로젝트 문서는 GitHub →"* 링크 포인터 1장만 둔다(내용 비유지, 링크만).

## 3. 공유 체계 — GitHub 네이티브 도구 매핑 ⭐

팀에 공유하는 건 RFC(개별 쟁점)만이 아니다. *큰 흐름*도 팀이 봐야 한다. Notion이 하던 PM 기능을 **GitHub 네이티브 도구로 전부 대체**한다 — surface가 하나라 sync 문제가 없다.

| PM 기능 | GitHub 도구 | 위치 |
|---------|------------|------|
| 오리엔테이션(큰 흐름)·연대기·규약 | 마크다운 (렌더링) | `docs/pm/` |
| 코드 spec | 마크다운 + PR 리뷰 | `docs/compliance/` |
| 의사결정 (ADR) | 마크다운 + PR 인라인 리뷰 | `docs/decisions/` |
| 열린 쟁점 | Issues (라벨로 유형·상태) | repo Issues |
| 상태 보드·칸반 | Projects | repo Projects |
| 큰 흐름 토론 | Discussions | repo Discussions |
| 변경 제안(RFC) | Pull Request | repo PR |
| 내 사고 원본·초안 | — (비공유) | Obsidian |

**핵심:** 변호사님은 이미 *오리엔테이션 패키지*(25·26·27·28·30·31 + 리딩가이드 32)를 만들어 뒀다. 재발명할 게 아니라 **`docs/pm/`으로 정제·이관**하면 되는 자산이다.

### 3-1. repo 문서 구조 (결정: 같은 repo `docs/`)

```
decipherhub/corner-store/
├── docs/
│   ├── pm/            ← 연대기+구현맵 · 운영 규약 (오리엔테이션·큰 흐름)
│   ├── compliance/    ← Legal-to-Technical Matrix · Element/Recipe spec sheet (노트 33 양식)
│   ├── decisions/     ← ADR (1결정 1파일 + index)
│   ├── architecture/  ← (기존) 책임 레이어
│   └── ROADMAP.md · MVP-v2-multi-venue.md   ← (기존)
├── [Issues]           ← 쟁점 등록부 = 라이브 이슈
├── [Projects]         ← 상태 보드
└── [Discussions]      ← 큰 흐름 토론
```

### 3-2. 안전 규칙 (필수)

- Claude·자동화는 **항상 브랜치 + PR**로 작업한다. `main` 직접 push 금지.
- 모든 변경은 변호사님 **PR 머지 승인** 후 반영 → 실수해도 복구 가능.
- 공유 문서는 볼트(Obsidian)에서 초안 → GitHub 발행 후 GitHub가 SoT.

### 3-3. 비개발자 팀원 (caveat)

보경 변호사 등이 GitHub 읽기·댓글에 불편하면 → Decipher Notion에 포인터 페이지 1장으로 보완하고, 핵심 검토자의 GitHub 사용 가능성을 한 번 확인한다.

## 4. 결정 프로세스 — 쟁점이 닫히는 길

쟁점 하나가 *발생해서 닫히기까지*의 파이프라인을 고정한다.

```
볼트 문제의식  →  쟁점 등록부(후보)  →  triage(고르기)  →  RFC/ADR 1장
   →  팀 토론(채널: GitHub/Notion)  →  결정  →  의사결정 등록부 기록  →  spec·코드 반영
```

**누가 결정하나**(쟁점마다 명시):
- 규제 단독(나) — 법적 판단이 핵심인 것 (예: F-04 추가 여부)
- 팀 합의 — 운영·우선순위가 걸린 것 (예: 긴급 정지 규칙)
- 개발팀 권한 — 구현 방식 (나는 *수정 요청서*로 전달만)
- 외부 변호사 자문 — 위임 Q-* (결정이 아니라 의견 수집)

## 5. 운영 리듬 (cadence)

| 주기 | 할 일 |
|------|-------|
| 매 작업 세션 종료 | 관련 등록부 갱신 + 볼트 sync 허브에 한 줄 |
| 매 회의 (주 1회, 화 20시) | 회의록 작성 → 액션 아이템을 등록부로 |
| 주간 | 연대기 구현맵 §5 상태 새로고침 + GitHub `docs/pm` 발행본 동기화 |
| 스프린트 (2주) | 노트 31의 스프린트와 정렬, 결정 세션 진행 |
