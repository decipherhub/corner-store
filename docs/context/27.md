# 우리 4-Layer는 "법률을 다루는 구조"입니다

> **이 문서의 한 줄**: 우리가 거래 측 compliance를 Element / Recipe / Manifest / Operator 네 layer로 나눈 건 임의의 기술 분할이 아닙니다. **각 layer는 법의 구성 요소에 정확히 대응합니다 — Element는 *구성요건*에, Recipe는 *규제*에, Manifest는 *자산*에.** 즉 우리 아키텍처는 법을 코드로 번역한 것입니다.
> 

---

## 0. 핵심 — 각 Layer는 법의 무엇에 대응하는가

```
┌──────────────────────────────────────────────────────────┐
│   Element   ↔   구성요건 (사실)                             │
│   Recipe    ↔   규제 (법률효과)                             │
│   Manifest  ↔   자산 (이 토큰에 적용되는 규제의 묶음)            │
│   Operator  ↔   집행자 (판단·승인·감시)                      │
└──────────────────────────────────────────────────────────┘
```

이 대응을 법률가의 사고 흐름으로 풀면:

| 법률가가 사건을 다루는 단계 | 대응 Layer | 무엇에 대응하나 |
| --- | --- | --- |
| ① **사실**을 인정한다 | **Element** | **구성요건 사실** (제재 대상인가·적격투자자인가·보유기간 충족했나) |
| ② **규제**의 요건에 포섭한다 | **Recipe** | **개별 규제** = 법률효과 1개 (Reg D 면제·Rule 144 safe harbor…) |
| ③ 이 **자산**에 어느 규제들이 적용되나 | **Manifest** | **자산(토큰)** — 그 토큰에 적용되는 규제의 묶음 |
| ④ 판단하고 집행한다 | **Operator** | **집행자** — 재량·승인·감시 |

**왜 이게 결정적인가:** "사실 → 규제 → 적용 대상 → 집행"이라는 법 적용의 골격은 *어느 나라, 어느 규제, 어느 법체계든 똑같습니다*. 그래서 우리 구조는 — 미국 증권법이든, 한국 자본시장법이든, EU 규제든 — **법률이 바뀌어도 layer를 새로 만들 필요가 없습니다.** 바뀌는 건 *내용*(어떤 Element·어떤 Recipe)이지, *구조*가 아닙니다. 이것이 우리 아키텍처의 일반성의 뿌리입니다.

---

## 1. 각 Layer의 법적 의미

### Element = 구성요건 사실

법조문의 요건은 더 못 쪼개는 *사실 단위*로 분해됩니다 — "적격투자자인가", "제재 대상인가", "보유기간을 채웠는가". Element는 **이 사실 하나를 판정하는 가장 작은 단위**입니다.

핵심: **사실의 어휘는 나라가 달라도 비슷합니다.** 제재·거주지·자격·보유기간·금액·신고 여부 — 어느 규제든 이걸 묻습니다. 그래서 Element는 *표준 라이브러리*이고, **함부로 늘리지 않습니다** (기존 것 재사용 우선, 신설은 최후 수단).

### Recipe = 개별 규제 (법률효과 1개)

Recipe의 단위를 "법률"이나 "조문"으로 잡으면 모호해집니다. 정답은 **법률효과**입니다 — "충족하면 무슨 효과가 생기나"가 한 문장으로 떨어지는 단위:

- Reg D 506(c) = "등록 면제가 성립한다"
- Rule 144 = "재판매가 적법해진다"
- §3(c)(7) = "펀드 등록 의무가 면제된다"

각각 *하나의 효과*입니다. 그리고 **한 거래에는 여러 효과가 동시에 걸립니다** — 그 효과들이 *전부 성립해야* 거래가 적법합니다. Element가 보수적인 것과 반대로, **Recipe는 활짝 열립니다** (새 규제 = 새 Recipe 등록).

### Manifest = 자산에 적용되는 규제 조율

"이 *토큰*에 어느 법들이 적용되나"는 토큰마다 다릅니다. Manifest는 **토큰별로 적용 법규를 식별·조율하는 layer**입니다 — 법률가가 "이 사안에 어느 법이 적용되나"를 정하는 단계. 발행인이 선언하고 DEX가 검증·승인하는 양측 구조라, 책임 경계가 여기 기록됩니다.

### Operator = 판단·집행

코드가 못 푸는 것(시세조종 판단·법률 해석·예외 개입)은 사람의 재량입니다. Operator가 판단하고, 그 결론을 권한 통제된 경로로 시스템에 넣습니다.

## 2. 임의의 규제를 어떻게 받아들이나 — 3개 질문

법을 4-Layer에 *배치*하는 게 §1이라면, **새 규제가 와도 그것을 어디에 어떻게 구현할지**가 자동으로 나와야 시스템이 완성됩니다. 그 답은 각 요건에 **3개 질문**을 던지는 것입니다:

| 질문 | 답의 종류 | 결정되는 것 |
| --- | --- | --- |
| **누가 판정하나?** | 코드가 직접 / 전문가 서명을 검증 / 사람이 판단 | 온체인 직접 vs claim 검증 vs 오프체인+게이팅 |
| **언제 작동하나?** | 거래 전 / 거래 시점 / 거래 후 | 사전 확인 vs 거래 차단 vs 사후 발동 |
| **무엇을 보고 판정하나?** | 이 거래만 / 누적 상태 | 단순 검사 vs 상태 추적(read+갱신) |

이 3개 좌표를 찍으면 — 어느 layer에 어떤 메커니즘으로 들어갈지가 *결정론적으로* 산출됩니다. 예:

- OFAC 제재 = 코드 직접 / 거래 시점 / 이 거래만 → 온체인 즉시 차단
- 적격투자자 = 전문가 서명 검증 / 거래 시점 / 이 거래만 → KYC 인증 확인
- holder 수 상한 = 코드 직접 / 거래 시점 / **누적 상태** → counter 읽고 갱신
- 시세조종 = **사람 판단** / 거래 후 / 누적 패턴 → 오프체인 판단 + 게이팅

## 3. 까다로운 규제도 새 layer 없이 흡수됩니다

법은 깔끔한 자동 차단만으로 되어 있지 않습니다. 세 가지 까다로운 유형이 있지만, 모두 *layer 신설 없이* 처리됩니다:

| 유형 | 예 | 처리 방식 (새 layer 없음) |
| --- | --- | --- |
| **코드가 못 푸는 것** | 시세조종·사기 (총체적 판단) | 시스템은 *표시(flag)*만, 사람이 판단, 결론이 권한 통제로 입력 |
| **누적으로 판정하는 것** | holder 수·거래량 한도 | **Manifest(범위·한도 지휘) + Element(실제 집계 연주) 협력** — Manifest는 카운터를 직접 건드리지 않음 |
| **규제끼리 충돌** | A국 공개 요구 vs B국 금지 | 충돌 규제는 같은 토큰에 동시에 안 붙이고, *시장(venue)을 나눠* 처리 |

특히 두 번째(누적)가 중요한데 — 처음엔 "별도 layer가 필요한가?" 우려가 있었지만, 검토 결과 **기존 Manifest와 Element가 절반씩 나눠 갖는 협력**으로 깔끔히 풀립니다. Manifest는 "어떻게 집계할지 지휘"하고, Element는 "실제로 집계"합니다. 지휘자는 악보를 직접 연주하지 않습니다.

→ **결론: 4-Layer 구조는 그대로, 까다로운 규제도 다 담깁니다.**

---

## 4. 그래서 — "확장에 열리고, 수정에 닫힌다"

규제·자산·국가가 변해도 **시스템 코드는 수정하지 않고, 등록으로 확장**합니다:

| 변하는 것 | 대응 |
| --- | --- |
| 새 규제·새 국가 | Recipe 신설 + Manifest 등록 |
| 새 토큰 | Manifest 등록 |
| 법 개정 | 버전 올림 (기존 거래는 보호) |
| 4-Layer 구조 자체 | **안 바뀜** |

한 가지 특수성: 여기서는 확장(새 규제 등록) 자체가 규제 관련 행위라 *심사 절차*를 거칩니다. "확장에 열려 있되, 열린 문에 문지기가 있다" — 이것이 규제 인프라에서의 올바른 확장성입니다.

---

## 한 문단 요약

> **Element는 구성요건, Recipe는 규제, Manifest는 자산에 대응합니다.** 우리 4-Layer는 법률가의 사고(사실 → 규제 → 적용 대상 → 집행)를 코드로 옮긴 구조입니다. 새 규제가 와도 "누가 판정·언제·무엇을 보고"라는 3개 질문에 답하면 구현 위치가 결정론적으로 나옵니다. 코드가 못 푸는 판단, 누적 집계, 규제 충돌 같은 까다로운 경우도 새 layer 없이 흡수됩니다. 그래서 *어떤 나라의 어떤 규제가 와도, 재설계 없이 등록으로 확장*합니다 — 이것이 우리가 만드는 "법률을 다루는 아키텍처"입니다.
